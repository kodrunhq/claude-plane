<task type="auto" tdd="true">
  <name>Task 2: DAGRunner + Orchestrator — DAG validation, execution engine, run lifecycle</name>
  <files>internal/server/orchestrator/dag_runner.go, internal/server/orchestrator/dag_runner_test.go, internal/server/orchestrator/orchestrator.go, internal/server/orchestrator/orchestrator_test.go</files>
  <behavior>
    - ValidateDAG detects cycles in step dependencies using Kahn's algorithm and returns error
    - ValidateDAG accepts valid DAGs (linear, diamond, parallel branches)
    - DAGRunner.Start launches all root steps (in-degree 0) concurrently
    - DAGRunner.OnStepCompleted unlocks dependent steps when all their prerequisites are done
    - DAGRunner.OnStepCompleted with failed step marks run as failed (on_failure=fail_run)
    - DAGRunner is thread-safe (concurrent OnStepCompleted calls don't corrupt state)
    - Orchestrator.CreateRun snapshots steps into run_steps and starts DAGRunner
    - Orchestrator.RetryStep resets target step and downstream skipped/cancelled steps to pending
    - Orchestrator.RetryStep re-evaluates DAG to execute the retried step
    - Orchestrator.CancelRun stops executing steps and marks remaining as cancelled
  </behavior>
  <action>
    Create `internal/server/orchestrator/dag_runner.go`:
    - `ValidateDAG(steps []store.Step, deps []store.StepDependency) error` — Kahn's algorithm, returns error if cycle detected
    - `StepExecutor` interface: `ExecuteStep(ctx context.Context, runStep store.RunStep) error` — called by DAGRunner to launch a step (creates a session on the target machine). In tests, use a mock that records calls and signals completion.
    - `DAGRunner` struct with fields: mu sync.Mutex, runID string, steps map[string]*store.RunStep, deps map[string][]string, dependents map[string][]string, inDegree map[string]int, executor StepExecutor, store store.JobStore, onRunComplete func(runID string, status string)
    - `NewDAGRunner(runID string, runSteps []store.RunStep, deps []store.StepDependency, executor StepExecutor, store store.JobStore, onComplete func(string, string)) *DAGRunner`
    - `Start(ctx context.Context)` — find root steps (in-degree 0), launch via goroutines
    - `OnStepCompleted(stepID string, exitCode int)` — mutex-protected. If exitCode != 0 and on_failure is fail_run: mark remaining steps as skipped, mark run as failed. If exitCode == 0: decrement dependents' in-degree, launch any that reach 0. If all steps completed, call onRunComplete with "completed".

    Create `internal/server/orchestrator/orchestrator.go`:
    - `Orchestrator` struct: mu sync.Mutex, activeRuns map[string]*DAGRunner, store store.JobStore, executor StepExecutor
    - `NewOrchestrator(store store.JobStore, executor StepExecutor) *Orchestrator`
    - `CreateRun(ctx context.Context, jobID string, triggerType string) (*store.Run, error)` — validate DAG, create run in DB, snapshot steps via store.InsertRunSteps, build DAGRunner, store in activeRuns, call Start
    - `RetryStep(ctx context.Context, runID string, stepID string) error` — reset target run_step to pending, reset downstream skipped/cancelled steps to pending, rebuild DAGRunner from DB state, launch
    - `CancelRun(ctx context.Context, runID string) error` — mark pending/running steps as cancelled, mark run as cancelled, remove from activeRuns
    - `OnStepCompleted(runID string, stepID string, exitCode int)` — routes to the correct DAGRunner; removes from activeRuns if run is done

    Tests should use a mock StepExecutor that tracks which steps were executed and in what order. Use channels or WaitGroups to synchronize concurrent step completions. Test cases:
    - Linear DAG (A -> B -> C): steps execute in order
    - Diamond DAG (A -> B, A -> C, B -> D, C -> D): A first, B and C parallel, D last
    - Cycle detection: A -> B -> C -> A returns error
    - Step failure: A succeeds, B fails, C (depends on B) is skipped, run status is failed
    - Retry: after failure, retry B, B succeeds, C executes
  </action>
